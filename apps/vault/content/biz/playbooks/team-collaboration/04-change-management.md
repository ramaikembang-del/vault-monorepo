# Change Management Process

**Last Updated:** 2025-12-27
**Owner:** Operations Team

---

## Overview

How to propose, review, and implement changes to products, processes, systems, or strategy at PWBI FnB Growth.

**Principle:** Change is good when intentional; chaos when uncontrolled.

---

## When to Use This Process

** USE for:**
- Major product changes (new features, deprecations)
- Process changes affecting multiple teams
- System architecture changes
- Pricing/packaging changes
- Organizational structure changes

** DON'T USE for:**
- Minor bug fixes
- Internal team process tweaks
- Experimental features (behind feature flags)

---

## Change Proposal Template

**Title:** [Brief description]
**Proposer:** [Name]
**Date:** YYYY-MM-DD
**Type:** [Product / Process / System / Strategy]

**Current State:**
- What exists today?
- What problem does it create?

**Proposed Change:**
- What will change?
- How will it work?

**Impact Analysis:**
- Who is affected? (customers, teams, systems)
- What's the effort? (engineering hours, cost)
- What are the risks?

**Alternatives Considered:**
- What else did we consider?
- Why did we reject alternatives?

**Decision:** [Approved / Rejected / Needs More Info]
**Approver:** [Name, Role]
**Implementation Date:** [Target date]

## Approval Workflow

### Small Changes (< 1 week effort)
1. **Proposer** creates change doc
2. **Relevant team lead** reviews
3. **Approval:** Team lead (24 hour SLA)
4. **Implementation:** Immediately after approval

### Medium Changes (1-4 weeks effort)
1. **Proposer** creates change doc
2. **Department head** reviews + comments
3. **Approval:** Department head + CPO/CTO (48 hour SLA)
4. **Communication:** Announce in all-hands before implementation
5. **Implementation:** Coordinated rollout

### Large Changes (> 1 month effort)
1. **Proposer** creates RFC (Request for Comments)
2. **Stakeholder review** (1 week comment period)
3. **Leadership review** (synchronous meeting)
4. **Approval:** CEO (or Board if major strategic change)
5. **Communication:** Written announcement + all-hands + Q&A
6. **Implementation:** Phased rollout with checkpoints

---

## Communication Requirements

**Before Implementation:**
- Announce change 1 week in advance (minimum)
- Explain "why" not just "what"
- Provide FAQ for common questions
- Offer training if needed

**During Implementation:**
- Daily status updates for large changes
- Quick wins/early feedback
- Surface issues immediately

**After Implementation:**
- Retrospective (what worked/didn't)
- Measure success metrics
- Iterate based on feedback
